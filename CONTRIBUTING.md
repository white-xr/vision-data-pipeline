# 贡献与提交规范

## 提交信息格式

提交信息统一使用类似 Conventional Commits 的格式：

```text
类型: 中文简短说明
```

示例：

```text
feat: 自动识别采集目录结构
fix: 修复误用非序列化权重提示
docs: 补充数据清洗使用说明
refactor: 重构数据集划分逻辑
chore: 更新项目配置文件
```

## 类型规则

- `feat`：新增功能
- `fix`：修复问题
- `docs`：文档修改
- `refactor`：代码重构，不改变功能
- `chore`：配置、依赖、脚本等杂项修改
- `test`：测试相关修改
- `style`：代码格式调整
- `perf`：性能优化

提交标题要求：

- 必须是中文
- 简短清楚
- 不要超过一行
- 不要写模糊描述
- 每次提交只对应一个相对独立的改动

## 自动提交规则

每完成一个相对独立、可运行、可验证的功能或修复后，先执行：

```powershell
git status
git diff
```

确认没有明显错误后再提交。

可以自动提交的情况：

- 新增一个完整小功能
- 修复一个明确 bug
- 更新一个独立脚本
- 补充或修正文档
- 调整配置文件
- 重构某个模块且功能不变

如果改动还没完成、代码无法运行、存在明显报错，不提交。

## 自动推送规则

提交成功后，如果当前仓库已经配置远程仓库，并且当前分支允许推送，可以自动执行：

```powershell
git push
```

遇到以下情况先停止并说明原因，不强行推送：

- 没有远程仓库
- 当前分支不明确
- 需要 `git push --force`
- 出现冲突
- 测试失败
- 有疑似数据集、权重文件、大图片、大视频被误加入
- 有账号、密钥、token、密码等敏感信息

## 文件管理规则

不要提交以下内容：

- `datasets/`
- `raw_data/`
- `captures/`
- `weights/`
- `runs/`
- `*.pt`
- `*.pth`
- `*.onnx`
- `*.engine`
- `*.mp4`
- `*.avi`
- `*.zip`
- `*.rar`
- `.env`

如果发现这些文件将被提交，先提醒维护者，并建议加入 `.gitignore`。

## 提交前检查

每次提交前至少检查：

```powershell
git status
```

如果项目里有测试或检查命令，也要优先运行，例如：

```powershell
python -m pytest
```

或者项目已有的检查脚本。
